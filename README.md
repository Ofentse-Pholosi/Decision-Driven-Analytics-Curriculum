# Decision-Driven Analytics for MT5 EA

### Bridging Business Analysis and Data Analytics Through a Self-Paced, Decision-First Curriculum

## Overview

This repository documents a **self-paced learning journey** designed to deliberately bridge **Business Analysis (BA)** and **Data Analytics**, using an MT5 automated trading system (Expert Advisor) as the applied domain.

Rather than presenting a polished, ‚Äúfinal‚Äù solution, this project is intentionally curated as a **transparent, step-by-step build-up of analytical capability**‚Äîfrom decision framing, through KPI design and data modeling, to dashboards and impact measurement.

The emphasis is not on claiming expertise upfront, but on **demonstrating how disciplined analytics thinking is developed in practice**.

## What This Project Is (and Is Not)

### What It Is

* A **self-paced analytics curriculum**, executed in public
* A practical demonstration of **upskilling in Business Analysis and Data Analytics**
* A decision-driven approach to analytics, where each step builds on the previous one
* A portfolio that shows *how* analytical thinking evolves, not just the end result

### What It Is Not

* A ‚Äúknow-it-all‚Äù trading system
* A black-box strategy claiming guaranteed profitability
* A dashboard-first or tool-driven analytics project

This repository intentionally exposes **assumptions, trade-offs, revisions, and learning progression**.

## Learning-First Design Philosophy

Most analytics portfolios only show outcomes.
This project shows **process**.

The work is structured as a sequence of modules, each producing a concrete artifact and tightening analytical discipline:

1. **Decision Framing** ‚Äì What decision is actually being made?
2. **Business-Aligned KPIs** ‚Äì How do we objectively judge success or failure?
3. **Data Modeling** ‚Äì How must data be structured to support those judgments?
4. **Analysis & Visualization** ‚Äì Only after the above are clear
5. **Impact Measurement** ‚Äì Did the decision change outcomes?

By treating the project as a curriculum rather than a showcase, the repository makes learning **auditable and reviewable**.

## Why This Project Exists

Many analytics efforts fail because they:

* Start with data instead of a decision
* Produce insights without ownership or action
* Optimise metrics without understanding risk
* Hide learning behind polished visuals

This project deliberately avoids those pitfalls by applying **business analysis discipline before analytics execution**.

The guiding question is:

> *Should a standardized MT5 EA configuration‚Äîrestricted to Step Index synthetic instruments and governed by strict risk controls‚Äîbe deployed for sustained capital allocation?*

Every artifact in this repository exists to support, test, or challenge that decision.

## Objectives

### 1. Demonstrate Decision Governance

* Clearly define decision ownership
* Document assumptions, constraints, and risks
* Avoid intuition-driven strategy changes

### 2. Build Business-Aligned Analytics

* Translate trading behaviour into decision-grade KPIs
* Prioritise capital preservation over raw returns
* Use leading and lagging indicators appropriately

### 3. Apply Analytics Engineering Discipline

* Model data explicitly for decision support
* Define correct analytical grain (trade, session, daily)
* Prevent common analytical errors through structure

### 4. Showcase Transparent Upskilling

* Show learning progression, not just outcomes
* Produce reviewable, portfolio-grade artifacts
* Make reasoning and trade-offs visible

## Contribution & Critique Guidelines

This repository is intentionally **open to review, critique, and discussion**.

Because the project is designed as a **self-paced learning curriculum**, external feedback is viewed as a core part of the learning process rather than an afterthought.

### What Kind of Feedback Is Welcome

Contributions and critiques are especially encouraged in the following areas:

* **Decision framing**

  * Are the business decisions clearly defined?
  * Are assumptions and constraints reasonable and explicit?

* **KPI design**

  * Do the metrics genuinely support the stated decision?
  * Are thresholds defensible and risk-aware?

* **Data modeling**

  * Is the chosen grain appropriate?
  * Are there risks of aggregation bias or metric distortion?

* **Analytical reasoning**

  * Are conclusions logically supported by the evidence?
  * Are alternative interpretations sufficiently considered?

* **Structure and clarity**

  * Is the progression from decision ‚Üí data ‚Üí insight understandable?
  * Can non-technical stakeholders follow the reasoning?

---

### How to Contribute

If you would like to contribute:

1. **Open an Issue**
